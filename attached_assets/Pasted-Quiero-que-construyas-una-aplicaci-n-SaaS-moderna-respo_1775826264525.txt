Quiero que construyas una aplicación SaaS moderna, responsive y mobile-first para España, orientada a autónomos y gestorías, que funcione como Sistema Informático de Facturación (SIF) en modalidad SOLO VERI*FACTU, preparada para integrarse con la AEAT y cumplir el RD 1007/2023, la Orden HAC/1177/2024 y las especificaciones técnicas publicadas por la AEAT que yo iré adjuntando al proyecto.

IMPORTANTE:
- No quiero un producto híbrido que permita alternar dinámicamente entre VERI*FACTU y NO VERI*FACTU.
- Quiero un producto concebido exclusivamente como VERI*FACTU.
- Debe quedar preparado para emitir posteriormente la Declaración Responsable del fabricante/productor del SIF.
- Debe ser multiempresa y multicliente, apto para gestorías y autónomos.
- Debe verse y funcionar muy bien en móvil, tablet y escritorio.
- Debe priorizar velocidad, simplicidad y UX muy superior a la aplicación gratuita de la AEAT.
- Debe tener una experiencia “crear factura en segundos”.
- Debe permitir escanear una factura o ticket/foto/PDF y usar IA para extraer datos, pero SIEMPRE dejando el resultado como borrador pendiente de validación humana antes de generar factura VERI*FACTU o remitir registro a la AEAT.
- Debe quedar preparado para portal de pruebas externas de la AEAT y también para producción, separando configuración por entorno.

OBJETIVO DE NEGOCIO:
Este producto tendrá dos grandes roles:
1) Autónomo / pyme: crear y enviar facturas muy rápido, consultar histórico, descargar PDF, ver estado AEAT, gestionar clientes y productos.
2) Gestoría: acceder a la información de varios clientes, consultar facturas remitidas, detectar incidencias, exportar información, supervisar estados, y actuar como representante cuando proceda.

DECISIÓN DE ARQUITECTURA:
Construye un SaaS web moderno con:
- Frontend: Next.js + TypeScript + Tailwind CSS + componentes accesibles y limpios.
- Backend: Next.js API routes o backend desacoplado en TypeScript si ves mejor separación.
- Base de datos: PostgreSQL con Prisma.
- Cola / trabajos: si es necesario, usar un sistema simple de jobs/reintentos para remisión a AEAT.
- Generación PDF: librería robusta y mantenible.
- QR: generación local conforme a las especificaciones de AEAT.
- OCR/IA: módulo independiente para ingestión de imágenes/PDF.
- Autenticación: email+password al principio, con diseño preparado para más adelante añadir certificado, Cl@ve, SSO o integraciones.
- PWA instalable en móvil, con aspecto de app.
- Arquitectura limpia, tipada, escalable y documentada.

REGLAS FUNCIONALES DEL PRODUCTO:
1. ROLES Y MULTI-TENANCY
- Debe existir modelo multi-tenant.
- Un usuario puede pertenecer a una o varias organizaciones.
- Tipos de organización:
  - autónomo/empresa final
  - gestoría
- Roles mínimos:
  - owner
  - admin
  - facturador
  - asesor/gestor
  - solo lectura
- Una gestoría puede gestionar múltiples obligados tributarios/clientes.
- Cada obligado tributario debe tener su propia “facturación” separada lógicamente, con cumplimiento independiente.
- En todo momento debe mostrarse claramente en pantalla para qué obligado tributario se está operando.

2. PRODUCTO SOLO VERI*FACTU
- El producto debe operar solo como VERI*FACTU.
- Cada factura válida debe generar su correspondiente registro de facturación de alta.
- El sistema debe preparar la remisión voluntaria a AEAT inmediatamente después de la generación del registro.
- Debe existir soporte para:
  - alta normal
  - subsanación
  - alta tras rechazo previo
  - anulación
  - rectificativa
- Diseña el dominio para soportar correctamente encadenamiento y trazabilidad.

6. ESCANEO DE FACTURAS / IA

- PDF
- escaneo desde cámara móvil

- correo reenviado en el futuro

La IA debe:
- detectar si el documento parece factura o no
- extraer proveedor/emisor si existe
- extraer destinatario si existe
- fecha
- número de factura
- base imponible
- IVA
- importe total
- líneas de producto si se pueden inferir

- CIF/NIF si aparece

MUY IMPORTANTE:
- No usar directamente el OCR/IA para remitir nada a AEAT sin confirmación humana.
- Todo documento escaneado debe crear un “borrador sugerido”.
- Mostrar claramente confianza por campo y campos dudosos.
- El usuario confirma, corrige o descarta.
- Debe quedar trazabilidad de origen: “generado manualmente” o “propuesto por IA”.
